Security Configuration

Designing BigFix's first-ever live compliance dashboard — from the ground up, through three design iterations, shaped by real platform constraints.

The Real Story
Before this product existed, BigFix had no live compliance view. A Security Admin who wanted to know their compliance posture had to navigate to each device, each device group, and each check individually — with no aggregated view, no filtering, no way to act without switching tools. Security Configuration changes that. Stage 1 is the first real-time compliance dashboard in BigFix's history.

The Problem

Security teams were flying blind.

BigFix had a compliance tool — Web Reports. But it wasn't built for how modern compliance teams work. There was no single place to see posture. No filtering. No path from identifying a failing check to fixing it without switching systems entirely.

What didn't exist — and what Security Configuration solves

1
No live compliance overview → Real-time landing page with KPI cards
Admins had to open each device, device group, and check individually. No aggregated environment-wide view existed anywhere in the platform.
2
No filtering capability → Global filter: 4 dimensions, multi-block OR logic
Admins managing specific regions or device groups had to cross-reference separate reports. There was no way to scope the view.
3
No saved or default views → Saved views with default auto-load
Every session started from scratch. Manual filter reapplication before any actual work could begin — every time.
4
No path from failure to action → Deploy directly from Security Checks grid
Admins had to leave the tool and switch to a separate system to act on compliance failures. Broken workflow, every time.
5
No per-checklist compliance detail → Single Checklist View: 5 charts + checks grid
Web Reports showed surface-level data. No OS segmentation, no device-level breakdown, no check-level investigation.

Key Design Decisions

Six calls that defined the product.

UX Principle · Filter Architecture

One filter per page — the principle that restructured everything

Early designs had a global filter at the top and a data table filter below on the same page. In cross-functional review, I flagged this as a fundamental UX problem: two filter layers on one page creates ambiguity — which one controls what? I introduced the principle: one filter per page. On the Landing Page, the global filter controls everything — no table filter. On the Single Checklist View, the global filter is absent entirely — the grid-level filter on the Security Checks table is the only one. This single principle restructured both pages and resolved confusion that would have surfaced in user testing.

→ Global filter removed from Single Checklist View. Grid filter handles scoping at that level. Zero ambiguity.
Performance-Driven Architecture

The two-page split was a performance decision, not an aesthetic one

The original design loaded 7 charts, a 132-row compliance table, and KPI cards simultaneously on a live platform with up to 300,000 managed endpoints. This made the page genuinely slow. The two-page split solved it by separating data loads: Landing Page loads lightweight data (KPI counts + checklist names) within 3 seconds. Single Checklist View loads heavy data (5 charts + Security Checks grid) within 20 seconds independently. Each SLA is achievable because neither page tries to do both. Performance SLAs in an enterprise product are design constraints, not engineering targets.

→ 3s SLA for Landing Page. 20s SLA for Single Checklist View. Neither blocked by the other.
Stakeholder Negotiation · KPI Cards

The Checklists KPI card: I pushed to remove it. When the PM kept it, I made the compromise work for users.

The landing page checklist grid already shows the count in its header — "Checklists (132)". A Checklists KPI card showing the same number is redundant. I raised this in review — it takes up prime visual real estate without adding information. The PM wanted it retained for consistency and overview feel. Rather than a disabled state (which implies something is broken) or an explanation tooltip, I differentiated it through two visual signals only: no chevron arrow, and the count in black instead of blue. The other three KPI cards use blue numbers and chevrons — interactive, navigating somewhere. The Checklists card sits among them but reads immediately as informational. PM's need met. User not misled.

→ 4 KPI cards. 3 interactive (blue + chevron). 1 informational (black, no chevron). No disabled state, no explanation needed.
Saved Views Placement

Saved Views lives inside the filter panel — not as a separate header element

A saved view is a saved filter configuration. Keeping Saved Views inside the filter panel makes that relationship explicit and immediate. An admin saving a view is saving a filter state — they should be in the same place. It also keeps the page header clean: only the filter icon and Export button at the top level. The Master Operator — who creates views for different teams — sees their view name the moment the filter panel opens, contextually tied to the filter they're about to edit.

→ Saved Views dropdown inside filter panel. Header stays lean. Relationship between view and filter is self-evident.
Terminology

"Remediate" became "Deploy" — because the action is enforcement, not reaction

In BigFix, Checks and Fixlets are the same thing. When an admin selects failing checks and clicks the action button, they are deploying a Fixlet to enforce a compliance check — a proactive enforcement action. "Remediate" implies reacting to a problem after it has already caused damage. "Deploy" reflects what is actually happening and how BigFix's model works. This was agreed in cross-functional review. It is a small word with large implication for how the product positions itself: as an enforcement tool, not a cleanup tool.

→ "Deploy" throughout the Security Checks grid. Consistent with BigFix's Fixlet deployment model and user mental model.
Naming & Navigation Placement

"Security Configuration" under Resolve — both decisions made through cross-functional alignment

Where a product lives in navigation tells users what it is for. Placing it under Resolve positions it as an action-oriented enforcement tool — you come here to identify failures and act on them, not just to generate reports. The name "Security Configuration" replaced the working name "Security Configuration Analytics." The word "Analytics" implied passive reporting. Removing it reinforces the product's identity as active compliance management. Confirming the name was a stakeholder alignment decision between the two PMs and myself — it unblocked repository creation, all UI strings, breadcrumbs, and the dev environment setup.

→ Under Resolve. Named "Security Configuration". Breadcrumb: Security Configuration → [Checklist Name].
Chart Strategy

5 charts. 2 removed. Every remaining one earns its place.

The earlier design had 7 charts — including "Bottom 10 by compliance %" and "Top 10 by compliance %" that compared checklists against each other. Once the design moved to a single-checklist view, these became meaningless — you can't rank checklists when you're already inside one. They were cut. Each of the 5 remaining charts exists because a specific persona needs a specific answer.

ChartThe question it answersPersona
Compliance donut What is the overall compliance % for this checklist right now? The most critical single number — answers "are we compliant?" before anything else is read. SC AdminMO
Compliance by Quartile How are devices distributed across compliance bands (0–24%, 25–49%, 50–74%, 75–99%, 100%)? Shows whether non-compliance is concentrated or spread — informs where to focus enforcement first. SC AdminIT Ops
Devices below Threshold For a threshold I define, how many devices fall below it — broken by OS? Lets admins ask "how many devices are below our 80% policy threshold" and prioritise by OS platform. SC AdminIT Ops
Check Results Of all checks in this checklist — how many are applicable, non-applicable, excepted, non-excepted, compliant, non-compliant? Surfaces whether exceptions are masking real failures. SC AdminMO
Compliance Distribution How many devices sit at each compliance percentage? Shows the shape — whether devices cluster near 100% (healthy) or spread across the range (systemic gaps throughout the environment). SC Admin

Charts 2–5 each have a table view icon that opens a data drawer — meeting WCAG 2.1 AA screen reader requirements. Chart 1 does not. A compliance donut showing 45% is already a single accessible number — a table view with one data point adds no accessibility value and would create false consistency.

Reflections

Honest notes.

Went well

Establishing the one-filter-per-page principle early gave me a consistent framework to defend structural decisions in review cycles. Principles are easier to align teams around than preferences — it made every filter-related conversation faster to resolve.

Went well

Treating performance SLAs as design constraints — not engineering problems — kept load time central to every layout decision. The two-page split, the KPI-cards-first hierarchy, and independent section loading all came from asking "what can we reliably show in 3 seconds" before asking "what should we show."

Went well

Using the three personas as a validation lens — not just a reference — gave every layout decision a user-grounded reason. When the PM wanted the Checklists KPI card retained, the persona framework gave me a way to negotiate within: keep it, but protect users from a false affordance.

Would do differently

I would push for at least one session with a real Security/Compliance Admin before wireframing. The personas were well-defined in the PRD but not user-validated. Direct access to someone using Web Reports today would have given me stronger evidence for the 5-chart hierarchy and made stakeholder conversations faster to close.

What's next

Stage 2 — Checklist Management — introduces create, edit, clone, and deploy flows for checklists. The Stage 1 grid patterns, filter architecture, and navigation structure were deliberately designed to scale into these flows. Stage 1 is not just a product — it is a design system foundation for everything that follows.
